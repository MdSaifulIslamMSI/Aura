# Login Provider Roadmap

## Current Provider State
The provider policy is codified in `server/config/authProviderPolicy.js`.

| Provider | State | Notes |
|---|---|---|
| Email/password | Active | Firebase-backed; backend session is authoritative. |
| Phone OTP | Active | Firebase phone OTP plus backend OTP fallback paths. |
| Google | Active | Firebase social provider. |
| Facebook | Active | Firebase social provider. |
| GitHub | Active | Firebase social provider. Requires a GitHub OAuth app client ID/secret saved in the Firebase GitHub provider configuration. |
| X | Active | Firebase social provider. |
| Microsoft | Active | Microsoft Entra app registration, Firebase `microsoft.com` provider, and CloudFront frontend exposure are enabled. Browser smoke confirms the Microsoft button is visible while Apple remains hidden. Existing-email collisions can be resolved by signing in with the current method, then linking Microsoft from Profile > Settings. |
| Apple | Ready when enabled | Frontend support exists behind `VITE_FIREBASE_ENABLE_APPLE_AUTH=true`; Apple developer/Firebase provider credentials are still required. |
| Enterprise OIDC | Design required | Do only when an enterprise customer needs it. |
| Enterprise SAML | Design required | Do only when an enterprise customer needs it. |

## Activation Rules
1. Do not enable Apple in production until Apple Developer credentials, authorized domains, mobile redirect settings, and staging browser smoke are complete.
2. Keep the GitHub OAuth app callback URL pointed at `https://<firebase-auth-domain>/__/auth/handler`; GitHub client ID/secret values belong in Firebase Authentication provider settings, not frontend env files.
3. Keep `VITE_FIREBASE_ENABLE_MICROSOFT_AUTH=true` in production builds while the Firebase `microsoft.com` provider remains enabled; repeat browser smoke after provider or domain changes.
4. Keep enterprise OIDC/SAML out of the consumer login UI until tenant routing, IdP metadata lifecycle, logout semantics, and account-linking behavior are designed.
5. Backend session sync remains the source of truth for user role and assurance state regardless of provider.

## GitHub OAuth Setup
GitHub OAuth App client IDs and client secrets are generated by GitHub's Developer settings flow; GitHub CLI can verify account access, but it does not expose a supported OAuth App creation command/API. Do not commit these values and do not add the secret to `VITE_*` frontend env files.

Use these public registration values:

```text
Application name: Aura Marketplace Auth
Homepage URL: https://aurapilot.vercel.app
Authorization callback URL: https://billy-b674c.firebaseapp.com/__/auth/handler
```

After GitHub shows the client ID and generated client secret:

1. Open Firebase Console > Authentication > Sign-in method > GitHub.
2. Enable GitHub and paste the GitHub OAuth App client ID and client secret.
3. Save the provider, then smoke-test the GitHub button on the hosted frontend.

Useful local checks:

```powershell
gh auth status
gh repo view MdSaifulIslamMSI/Aura --web
```

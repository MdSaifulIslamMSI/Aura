# Vercel Staging Frontend

Current source of truth:

- Backend staging is the AWS EC2 Docker Compose stack under `/aura/staging`.
- Frontend staging is currently the Docker-hosted static frontend on the same AWS staging origin as the backend: `http://ec2-13-201-55-118.ap-south-1.compute.amazonaws.com`.
- Vercel custom staging and branch-scoped Preview env writes are blocked for this project today, so Vercel Preview remains frontend-only evidence unless it passes the same frontend staging smoke contract.
- Production Vercel preview URLs remain frontend previews only unless a deployment passes the frontend staging smoke contract.
- A staging frontend must never route backend paths to production CloudFront or any production API origin.

## Decision Table

| Vercel capability | Staging mode | Expected result |
| --- | --- | --- |
| Pro/Enterprise with admin token | Custom environment `staging` | `vercel deploy --target=staging`, staging env vars, smoke required |
| Hobby/free or custom env unavailable | Preview branch fallback | Branch-scoped Preview env for `staging`, generated Vercel URL, smoke required |
| No domain permission | Generated deployment URL | Allowed if smoke proves AWS staging backend |
| No usable Vercel auth | Code/docs/checks only | Fail closed with `docs/staging-vercel-blocker-report.md` |
| Vercel blocked but AWS staging is live | Docker-hosted AWS frontend | Serve static `app/dist` from `nginx:alpine` and proxy backend paths to AWS staging |

## Required Staging Variables

The frontend staging deployment must use these non-secret values:

```sh
STAGING_FRONTEND_URL=http://ec2-13-201-55-118.ap-south-1.compute.amazonaws.com
STAGING_API_BASE_URL=http://ec2-13-201-55-118.ap-south-1.compute.amazonaws.com
STAGING_HEALTH_URL=http://ec2-13-201-55-118.ap-south-1.compute.amazonaws.com/health
STAGING_SSM_PREFIX=/aura/staging
```

Vercel receives these public build/deploy values for the staging deployment:

```sh
VITE_API_URL=/api
AURA_BACKEND_ORIGIN=$STAGING_API_BASE_URL
NEXT_PUBLIC_API_BASE_URL=$STAGING_API_BASE_URL
NEXT_PUBLIC_APP_ENV=staging
NEXT_PUBLIC_ENVIRONMENT=staging
NEXT_PUBLIC_STAGING_BACKEND_URL=$STAGING_API_BASE_URL
```

The deployment-specific Vercel config generated by `npm run staging:vercel:autopilot` rewrites backend paths to the AWS staging backend before smoke runs. If Vercel blocks custom staging or Preview env writes, use the Docker frontend path instead:

```sh
npm run staging:frontend:docker
STAGING_FRONTEND_URL=$STAGING_API_BASE_URL npm run smoke:staging:frontend
```

The Docker frontend builds with `VITE_API_URL=/api`, serves static files from a private `nginx:alpine` container bound to `127.0.0.1:8080`, and keeps `/api`, `/health`, `/uploads`, and `/socket.io` on the AWS staging backend through host Nginx.

## Commands

```sh
npm run staging:vercel:autopilot
npm run staging:frontend:docker
npm run smoke:staging:frontend
npm run scan:prod-fallbacks
```

The autopilot writes `.staging/vercel-staging-result.json` locally. That file is intentionally ignored.
Frontend smoke no longer reads local `.staging` deployment output as a fallback; CI and local runs must pass explicit `STAGING_FRONTEND_URL`, `STAGING_API_BASE_URL`, and `STAGING_HEALTH_URL` values. Vercel automation bypass secrets are generated for the run and are not cached in `.staging`.

## Safety Rules

- Do not overwrite production Vercel env vars for staging.
- Do not treat a generic Vercel Preview URL as backend staging.
- Do not pass frontend staging smoke unless `/health` through the frontend returns `env: staging` and `ssmPrefix: /aura/staging`.
- Do not allow frontend bundles, CSP, redirects, or backend route responses to contain production backend origins.
